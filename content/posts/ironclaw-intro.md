## **前言**

OpenClaw 作为一款流行的个人 AI 助手框架，在生产环境中曾发生过事故。这给我们一个重要的启示：**安全性和可靠性是生产级 Agent 框架的核心要求**。

IronClaw 作为 OpenClaw 的 Rust 原生重新实现，从架构设计层面就将安全、隐私、可靠性作为第一优先级。本文基于 IronClaw 的源码架构，深入解析其核心技术实现。

------

## **二、核心设计差异**

### **2.1 数据库层：全量数据持久化**

与其他 Claw 框架不同，**IronClaw 摒弃了文件系统**，所有用户数据均基于 PostgreSQL 存储：

- 会话历史
- 记忆（Memory）
- 密钥（Secret）
- 配置信息

**这种设计带来的优势：**

- 天生的数据安全，可利用现有完善的数据库管理系统
- 确保数据安全、高可用
- 便于备份、迁移和审计
- 统一的数据访问接口，简化了系统复杂度

------

### **2.2 密钥管理：企业级安全设计**

IronClaw 的密钥管理是其安全架构的核心亮点：

```
 ┌─────────────────────────────────────────┐
 │           macOS 钥匙串服务               │
 │         (存储 Master Key)               │
 └─────────────────┬───────────────────────┘
                   │ 启动时动态获取
                   ▼
 ┌─────────────────────────────────────────┐
 │         IronClaw 密钥服务               │
 │    (动态解密数据库中的加密密钥)          │
 └─────────────────┬───────────────────────┘
                   │
                   ▼
 ┌─────────────────────────────────────────┐
 │            PostgreSQL                   │
 │   (存储使用主密钥加密过的各种 Secret)    │
 └─────────────────────────────────────────┘
```

**关键特性：**

- 每次启动时动态获取 macOS 上的主密钥
- 数据库中存储的密钥均经过主密钥加密
- **只能通过 IronClaw 的密钥服务获取密钥**，其余方式均无法获取
- 即使数据库被泄露，攻击者也无法解密其中的密钥

这种设计借鉴了企业级密钥管理的最佳实践，将密钥的存储和使用完全分离，确保密钥在整个生命周期中都处于加密状态。

------

### **2.3 LLM Provider：策略层设计**

IronClaw 提供了多种 LLM Provider，并在之上构建了完整的策略层：

- **重试机制（Retry）**：自动重试失败请求，提高可用性
- **熔断机制**：防止级联故障，保护系统稳定
- **智能路由**：根据请求类型选择最优 Provider
- **回退机制**：当主 Provider 不可用时的降级方案

这种设计使得系统能够灵活应对各种 LLM 服务的波动，保证用户体验的稳定性。

## **三、Tools 系统设计**

### **3.1 Tool 接口定义**

Tools 在 IronClaw 中是一个标准化接口，每个 Tool 都包含以下属性：

- 名称和描述
- 参数格式定义
- 执行能力
- 是否需要用户明确同意

这种标准化设计使得 Tool 的扩展和管理变得非常简单。

### **3.2 内置 Tools**

**基础 Tools（4 个）：**

- `echo`：回显测试
- `time`：时间查询
- `json`：JSON 处理
- `http`：HTTP 请求

**其中 HTTP Tool 内置了完善的安全机制：**

- 完善的超时和重定向策略
- 校验下载文件的目录，防止目录穿越
- 强制 HTTPS，限制可访问的域名
- 凭证注入管理
- 密钥泄漏检测
- 响应内容处理
- 支持模拟请求响应

**开发相关 Tools（默认未启用）：**

- `shell`：执行 shell 命令
- `readfile`：读取文件
- `writefile`：写入文件
- `listdir`：列出目录
- `applypatch`：修改文件。
- `BuildSoftwareTool`：内置 Prompt，调用大模型和其他 Tools 生成应用

## **四、Extension 系统：WASM 沙箱**

### **4.1 扩展架构**

IronClaw 的扩展系统基于 WASM（WebAssembly）实现，主要流程如下：

1. 创建 WASM 运行时环境
1. 从目录导入 WASM 模块
1. 扫描目录下的 WASM 文件及其对应的能力配置文件
1. 将 WASM 模块包装成标准 Tool
1. 运行时动态注入能力

### **4.2 安全隔离**

WASM 沙箱设计带来了以下安全优势：

- 工具和 Channel 扩展都基于 WASM 实现，提供更好的隔离性
- **默认没有系统调用权限**，实现极致安全
- 能力（Capabilities）在运行时动态注入，最小权限原则
- 使用接口定义语言（WIT）规范 Host 和 WASM 模块之间的交互

这种设计确保了即使第三方扩展存在恶意代码，也无法对宿主系统造成危害。

------

## **五、Channel：用户沟通渠道**

Channel 模块主要负责与用户进行沟通，支持多种通讯协议的接入。它是 Agent 与外部世界交互的窗口，负责消息的接收和发送。

并且第三方的Channel实现也可以运行在WASM沙箱中，确保主进程的安全。

------

## **六、Agent：核心调度模块**

Agent 是整个系统的核心模块，包含三个核心功能：

1. **渠道对接**：与 Channel 打通，接收和发送消息
1. **路由（Router）**：识别用户意图并进行路由
1. **调度（Scheduler）**：负责任务的调度和执行

Agent 连接了 Channel、LLM 和 Tools，是整个系统的指挥中心。

## **七、安全架构总结**

用户使用IronClaw时，威胁主要来自于：

- 第三方Tools。
- 注入的Prompt。
- LLM异常调用工具。

### **7.1 Agent主体**

- 主体基于 Rust 实现
- 提供极高的内存安全和性能保证
- 编译时检查避免了大量运行时错误
- 代码开源，其主体逻辑安全可信。

### **7.2 沙箱隔离**

- 工具和 Channel 扩展都基于 WASM 实现
- 提供更好的隔离性
- 默认没有系统调用权限，极致安全

### **7.3 密钥管理**

- 密钥动态注入，工具、LLM、Channel 本身都不能直接接触到密钥
- 密钥泄漏检测，针对非预期的请求头密钥会进行精细拦截
- 主密钥存储于操作系统钥匙串服务，应用无法直接访问

### **7.4 数据存储安全**

- 底层数据存储全部依赖 PostgreSQL
- 数据库中存放的密钥也是加密过的
- 需要依赖用户提供的 Master Key 来进行动态解密
- 在 macOS 上，Master Key 使用系统钥匙串应用进行存储

### **7.5 一次github工具调用流程**

1. LLM识别到需要调用github工具，填好参数之后，启动wasm沙箱，执行github的wasm模块。

1. github的wasm模块发起http请求，访问github.com。注意：这里不是直接在wasm调用的http client，而是调用的ironclaw专门注册的一个http工具。

1. http工具的参数是url、header、body等这些参数。其代码在ironclaw主仓库中。在ironclaw的http工具中会执行secret泄漏扫描以及注入。

   关于Secret注入多解释一下：

   1. IronClaw在启动时需要调用Mac的钥匙串应用，此时会需要你输入Mac密码，然后IronClaw才能获取到主密钥。
   1. 拿到主密钥之后，IronClaw通过主密钥可以解密拿到具体的某个需要被注入的Secret。整体确保了Secret的安全性。

   下面一个github工具的限制策略。

   ```
    "capabilities": {
        "http": {
          "allowlist": [
            {
              "host": "api.github.com",
              "path_prefix": "/",
              "methods": [
                "GET",
                "POST"
              ]
            }
          ],
          "credentials": {
            "github_token": {
              "secret_name": "github_token",
              "location": {
                "type": "bearer"
              },
              "host_patterns": [
                "api.github.com"
              ]
            }
          },
          "rate_limit": {
            "requests_per_minute": 60,
            "requests_per_hour": 3600
          }
        },
   ```

   这个策略表明了，这个工具只能访问api.github.com，并且依赖github_token这个secret。secret是由ironclaw存储在数据库中的，tools本身并不会接触到。

1. 最终的http调用是ironclaw中的http工具发出的。

## **八、总结与展望**

IronClaw通过以下设计实现了企业级的安全保障：

| **设计维度** | **实现方案**  | **安全收益**                |
| ------------ | ------------- | --------------------------- |
| 编程语言     | Rust 原生实现 | 内存安全、性能优异          |
| 数据存储     | 全数据库存储  | 摒弃文件系统，利用成熟 DBMS |
| 扩展隔离     | WASM 沙箱     | 工具隔离、无系统调用        |
| 密钥管理     | 分层密钥管理  | 动态解密、泄漏检测          |
| 服务稳定     | LLM 策略层    | 重试、熔断、智能路由        |

未来演进展望：

1. 更加统一和完善的工具生态：
   1. 当前IronClaw还处于快速迭代期，WASM工具生态还不完善，不如openclaw丰富。不过这点随着项目演进会逐步成熟。
   1. 这里其实还衍生出了另一个问题：如何提供一个基于wasm的标准工具运行时？
   1. Tools标准化：现在的tools有很多协议，MCP、各种Claw自己定义的Tools甚至E2B。
1. 更加云原生：
   1. 云厂商提供密钥管理工具，能够更加安全的实现密钥轮转和注入。目前服务网格/AI网关等技术可以做到非WASM部分的身份认证和密钥动态注入，但是WASM部分有待标准形成。
   1. 存储工具：支持数据库、OSS等多种类型的存储供agent和tools使用。
   1. 结合Wasm运行时和服务网格进行统一流量管控和可观测。

IronClaw 的安全架构为企业级 AI Agent 提供了一个可靠起点，但要真正适应云原生环境，还需要在工具标准化和云服务集成方面继续演进。
