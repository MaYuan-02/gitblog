## 背景

传统 API 请求通常只有几 KB。AI 推理场景不一样——多轮对话会累积历史消息，RAG 会注入整篇文档作为上下文，多模态请求中一张 base64 编码的图片就是几 MB。在 RAG 和多模态场景下，单个请求达到几 MB 甚至 10MB 以上并不罕见，而且随着长上下文模型和多模态的普及，请求体积还在持续增长。

但网关要做的事没变：从请求中提取 model 字段，做路由、限流、鉴权。问题在于，OpenAI 的 API 标准把 model 放在了请求 body 的 JSON 里，而不是 URL 或 header 中。这意味着网关必须先读 body 才能确定路由目标。在拿到 model 之前，请求不能转发给 upstream——因为连发给谁都不知道。坦率地说，这不是一个好的 API 设计，但 OpenAI 格式已经成为事实标准，vLLM、TGI、Ollama 都遵循它，我们只能适应。

推理网关基于 Envoy，通过 ExtProc（External Processing）实现 body 解析和路由决策。

## 第一版：Go 全量解析

第一版用 Go 写，逻辑很直接：收完所有 body chunks，拼起来，json.Unmarshal 反序列化，取出 model。

用 10MB 的请求压测（2C2G，1000 个请求）：

| 并发 | 成功数   | 平均延迟 | P90 延迟 | 内存   |
| ---- | -------- | -------- | -------- | ------ |
| 10   | 1000     | 563ms    | 649ms    | 331MB  |
| 20   | 1000     | 1.15s    | 1.46s    | 632MB  |
| 50   | 1000     | 2.85s    | 3.54s    | 1.48GB |
| 100  | 全部失败 | -        | -        | OOM    |

50 并发吃掉 1.48GB 内存，100 并发 OOM。每个请求都把 10MB JSON 完整反序列化为 Go 对象，50 个并发就是 500MB 原始数据加上对象开销和 GC 碎片。为了一个 model 字段，把整个 10MB body 都加载到了内存里。

## 核心思路：只解析你需要的

model 字段在 JSON 里不到 100 个字节。在绝大多数实际场景中，OpenAI SDK 和主流框架序列化时都会把 model 排在前几个字段（JSON 规范中 key 是无序的，不能保证所有情况都如此，但实际生产流量中 model 在前面是绝对多数）。

那为什么要为了 100 个字节解析 10MB？

- **全量解析**：收完 10MB → 反序列化 → 取 model
- **流式解析**：读前几百字节 → 找到 model → 停止 → 剩余数据直接透传

## 第二版：Rust 流式解析

为了探索流式解析的性能上限，我们用 Rust 重新实现了 ExtProc。选 Rust 不是为了替换 Go，而是想做一次极致的性能测试。Go 的 json.Decoder 也能做流式解析，但跳过字段时仍需分配内存。Rust 的 serde 提供了 IgnoredAny，跳过一个 9.99MB 的 JSON 数组时只做语法扫描，零分配。

### 三个关键实现：

1. **serde Visitor 模式**：自定义 Visitor 遍历 JSON 顶层 key，遇到 model 提取值，其余字段用 IgnoredAny 跳过，不构建对象，不分配内存。
2. **ChannelReader 适配器**：ExtProc 通过 gRPC stream 分 chunk 收 body，serde 需要连续字节流。我们写了个适配器把 channel 包装成 Read 接口，避免拼接完整 body。
3. **零拷贝回放**：已消费的 chunks 原样保存，解析完后按顺序回放给 Envoy，后续 chunks 直接透传。全程不拼接、不复制。

### 压测结果

所有测试使用 10MB 的 OpenAI 格式请求，1000 个请求。选择 10MB 是为了放大问题、充分体现不同方案的差异。

#### 内存对比（2C2G）：

| 并发 | Go 全量解析 | Rust 流式（model 在前） | 降幅  |
| ---- | ----------- | ----------------------- | ----- |
| 50   | 1.48GB      | 8.9MB                   | 99.4% |
| 100  | OOM         | 22MB                    | -     |
| 200  | OOM         | 44MB                    | -     |

#### 延迟对比（2C2G）：

| 并发 | 直连（无 ExtProc） | Go 全量解析 | Rust 流式（model 在前） |
| ---- | ------------------ | ----------- | ----------------------- |
| 20   | 62ms               | 1.15s       | 344ms                   |
| 50   | 0.7s               | 2.85s       | 0.866s                  |
| 100  | 1.5s               | OOM         | 1.77s                   |
| 200  | 2s                 | OOM         | 3.73s                   |

#### CPU 对比（2C2G）：

| 并发 | Go 全量解析 | Rust 流式（model 在前） |
| ---- | ----------- | ----------------------- |
| 50   | 2C 打满     | 0.33C                   |
| 100  | OOM         | 0.51C                   |
| 200  | OOM         | 0.52C                   |

#### 最差情况：model 在 JSON 末尾

| 并发 | 平均延迟 | P90 延迟 | 内存  | CPU |
| ---- | -------- | -------- | ----- | --- |
| 50   | 3.9s     | 4.2s     | 500MB | 2C  |
| 100  | 7.8s     | 8.3s     | 850MB | 2C  |
| 200  | 15s      | 17s      | 1.5GB | 2C  |

对于需要全量扫描的场景，流式解析的绝对性能要弱于直接全量解析。由于此次测试中全部用的流式解析，所以model在最末尾是Rust延迟略低于Go。

实际生产中，可以采取流式和全量解析结合的方式，减少流式解析所有Chunk的场景。

## 总结

### 代理层处理大 body 的原则

能不解析就不解析，必须解析时流式优先，只提取需要的字段。10MB 的 body 中 99.99% 的内容与路由决策无关。随着 AI 请求体积持续增长——长上下文窗口从 4K 扩展到 128K 甚至 1M，多模态从单张图片到多图、音频、视频——高效处理大 body 不再是优化项，而是网关的基本能力。

### 关于语言选择

Go 同样可以做流式解析，普通场景够用。Rust 的优势在高并发条件下更明显——零分配跳过和无 GC 让它在最差情况下也保持可控，带来了显著的稳定性提升。选什么取决于团队情况，流式解析的思路本身是语言无关的。

从 2G OOM 到 44MB，两个数量级的差距，来自一个简单的问题：你真的需要解析整个 body 吗？
